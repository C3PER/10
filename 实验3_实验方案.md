## 五、阶段二：模型训练（Python/PyTorch）

### 5.1 模型选型：BC-ResNet

按照实验指导书要求，选用 **BC-ResNet**（Broadcasted Residual Learning for Efficient Keyword Spotting）作为命令词识别网络。

**BC-ResNet 的核心设计思想：**

BC-ResNet 是专为音频关键词识别（Keyword Spotting, KWS）任务设计的轻量级深度网络，其主要创新点包括：

| 特性 | 说明 |
|------|------|
| **Log-Mel 谱输入** | 以 Log-Mel 频谱图作为输入，适配音频分类任务 |
| **通道广播残差（Broadcast Residual）** | 通过时域平均池化 + 1×3 膨胀卷积 + Sigmoid 门控，实现跨分支特征复用，大幅降低参数量和计算量 |
| **双维度残差** | 将传统空间残差扩展为"频率-时间"双维度残差连接 |
| **深度可分离卷积** | 在频率维度用 3×1 深度卷积优化频谱局部特征提取 |
| **膨胀卷积（Dilated Conv）** | 在时间维度上用膨胀卷积扩大感受野，捕捉语音的长时依赖（如音节级特征） |
| **多尺度特征融合** | 通过不同膨胀率的残差块级联，实现多尺度时间特征融合 |
| **8bit 量化支持** | 网络参数支持 8bit 整型量化，便于在嵌入式 DSP 上高效部署 |

**与其他架构的比较：**

相比指导书中讨论的其他架构（DS-CNN、MobileNet、ResNet 等），BC-ResNet 在参数量（~200K）、计算效率和音频识别准确率之间取得了最佳平衡，且指导书已提供完整的 C 语言推理代码样例和预训练权重。

### 5.2 特征提取

按照指导书样例代码的参数配置：

| 参数 | 值 | 说明 |
|------|-----|------|
| 采样率 (FS) | 16000 Hz | 与数据集一致 |
| 音频长度 (SAMPLES_NUM) | 16000 点 | 1 秒音频 |
| 帧长 (FRM_LEN) | 160 点 | 10ms |
| 窗长 (WIN_SIZE) | 480 点 | 30ms（包含前后各 160 点的上下文） |
| FFT 点数 (FFT_LEN) | 512 | 2 的幂次 |
| 频率点数 (FREQ_NUM) | 257 | FFT_LEN/2 + 1 |
| Mel 滤波器数 (MELS_NUM) | 40 | 指导书默认配置 |
| 窗函数 | Hann 窗 | `0.5*(1 - cos(2πn/WIN_SIZE))` |
| 帧数 W | ~101 | SAMPLES_NUM / FRM_LEN + 1 |

**特征提取流程：**

```
原始音频 (16000 点)
  → 对称 padding（前后各 WIN_SIZE/2 = 240 点）
  → 分帧：帧长 480，帧移 160，得 ~101 帧
  → 每帧加 Hann 窗
  → 512 点 FFT
  → 计算功率谱（实部² + 虚部²）
  → Mel 滤波器组（40 个三角滤波器，频率范围 0-8kHz）
  → log(⋅ + 1e-6)
  → 输出 Log-Mel 谱：维度 40 × W（W ≈ 101）
```

### 5.3 BC-ResNet 网络结构

按照指导书表 3.4.6 的概要设计和 C 代码样例，BC-ResNet 的网络结构如下：

```
输入: Log-Mel 谱 (1 × 40 × W), W ≈ 101
  │
  ├─ Conv2d(1→16, kernel=5×5, stride_h=2) → BN → ReLU
  │   输出: 16 × 20 × W
  │
  ├─ BC-ResBlock ×2 (Stage 0, 膨胀率=1)
  │   [16→8, freq_stride=1] + [8→8, freq_stride=1]
  │   输出: 8 × 20 × W
  │
  ├─ BC-ResBlock ×2 (Stage 1, 膨胀率=1)
  │   [8→12, freq_stride=2] + [12→12, freq_stride=1]
  │   输出: 12 × 10 × W
  │
  ├─ BC-ResBlock ×4 (Stage 2, 膨胀率=2)
  │   [12→16, freq_stride=2] + [16→16, freq_stride=2]×3
  │   输出: 16 × 5 × W
  │
  ├─ BC-ResBlock ×4 (Stage 3, 膨胀率=4)
  │   [16→20, freq_stride=3] + [20→20, freq_stride=3]×3
  │   输出: 20 × 5 × W
  │
  ├─ DWConv(5×5, padding=2) → BN → ReLU
  │   输出: 20 × 1 × W
  │
  ├─ Conv2d(20→32, kernel=1×1) → BN → ReLU
  │   输出: 32 × 1 × W
  │
  ├─ Global AvgPool (时间维度)
  │   输出: 32 × 1 × 1
  │
  └─ Conv2d(32→12, kernel=1×1)  ← 分类器
      输出: 12 (logits)
```

**BC-ResBlock 内部结构：**

每个 BC-ResBlock 包含两条并行的处理路径：

```
输入 x (C_in × H × W)
  │
  ├─ 主路径 (Main Path):
  │    ├─ Conv2d 1×1 (C_in→C_out) → BN → ReLU
  │    └─ DWConv 3×1 (仅频率维度) → BN → ReLU → 通道扩展(×5)
  │
  └─ 广播残差路径 (Broadcast Residual Path):
       ├─ AdaptiveAvgPool (频率维度 → 1)
       ├─ Conv2d 1×3 (膨胀卷积, 扩张率=dilation, 仅时间维度) → BN → Sigmoid (门控)
       └─ Conv2d 1×1 (C_out→C_out)
       │
       └─ 与主路径相加 → ReLU → 输出
```

BC-ResBlock 的关键设计：
- **频率维度降维**：通过 AdaptiveAvgPool 将频率维度压缩为 1
- **时间膨胀卷积**：用膨胀率 1/2/4 的 1×3 卷积捕获不同尺度的时间依赖
- **Sigmoid 门控**：产生 [0,1] 的注意力权重，选择性增强重要时间帧
- **残差相加**：门控后的广播特征与主路径特征相加，实现跨通道特征复用

## 七、阶段四：权重导出与 C 代码推理实现

### 7.1 权重导出

从 PyTorch 模型中提取所有权重参数，导出为 C 数组。BC-ResNet 需要导出的权重类型如下：

```python
# 需要导出的参数：
# 1. cnn_head_0_weight[16][1][5][5]  — 入口 Conv2d 权重 (OIHW 格式)
# 2. cnn_head_1_weight[16]           — 入口 BN weight
# 3. cnn_head_1_bias[16]             — 入口 BN bias
# 4. cnn_head_1_running_mean[16]     — 入口 BN running_mean
# 5. cnn_head_1_running_var[16]      — 入口 BN running_var
#
# 每个 BC-ResBlock 包含：
#   - f2_0: 1×1 Conv weight → BN weight/bias/mean/var
#   - f2_1: 3×1 DWConv weight → BN (ssnorm) weight/bias/mean/var
#   - f1_0: 1×3 DilatedConv weight → BN weight/bias/mean/var
#   - f1_1: 1×1 Conv weight
#
# 最后还有：
#   - DWConv 5×5 weight
#   - Conv2d 20→32 weight → BN weight/bias/mean/var
#   - Classifier Conv2d 32→12 weight + bias
#
# 格式转换注意：
#   - PyTorch Conv2d: (O, I, H, W) → C: (O, I, H, W)
#   - PyTorch BN: (C,) → C: (C,)
#   - 直接写出为 .c 文件中的 double 数组
```

导出脚本输出：
- `kws_weights.h`：所有权重数组的 extern 声明 + 维度宏定义
- `kws_weights.c`：权重数据（等价于指导书的 `table.c`）

### 7.2 C 代码文件结构

按照指导书样例的实际组织结构：

```
Code/User/
├── user_kws.h          # KWS 模块头文件（参数宏定义、接口声明）
├── user_kws.c          # 顶层接口 + Mel谱特征提取 + 网络推理（单体实现）
├── table.c             # 权重数据 + 全局宏定义（FRM_LEN, WIN_SIZE, 网络通道数等）
└── (其他外设驱动文件)
```

指导书将特征提取、网络推理、主函数放在一个文件中（见 `exp3_body.txt`），通过 `#include "table.c"` 引入权重。实际开发时可以根据需要做一定程度的模块化拆分，但要保证与指导书样例的兼容性。

### 7.3 接口设计

## 八、阶段五：DSP 平台部署与实时识别

### 8.1 代码移植

将 PC 端验证通过的 C 推理代码集成到 CCS 工程中：
- 将特征提取和网络推理代码放入 `user_kws.c`
- 将权重数据作为 `table.c` 或独立权重文件加入工程
- 配置 CCS 编译器选项（建议使用 `-O3` 优化）

### 8.2 内存规划

| 数据类型 | 大小估算 | 存放位置 | 说明 |
|---------|---------|---------|------|
| 网络权重 | ~800KB | NAND Flash | 约 200K 个 float/double 参数 |
| 音频缓冲区 | ~32KB | L2 SRAM | 16000 short 采样点 |
| Hann 窗 | ~2KB | DDR2 SDRAM | WIN_SIZE=480 个 double |
| Mel 滤波器组 | ~10KB | DDR2 SDRAM | 257×40 个 double，初始化计算一次后不变 |
| 中间特征图 | ~1MB | DDR2 SDRAM | 运行时动态分配，最大层为 CHANNEL×W×H |

由于 C6748 的 L2 SRAM 有限（256KB），大的中间缓冲区必须放在 DDR2 SDRAM 中。

### 8.3 计算优化

- 使用 `float` 替代 `double`（C6748 支持硬件单精度浮点，可显著加速）
- 关键循环使用 `#pragma UNROLL` 展开
- Conv 和 BN 的循环合理安排访存顺序以提高 Cache 命中率
- 可选：使用 TI DSPLIB 中的 FFT 函数替代自编 FFT

### 8.4 外设集成

| 外设 | 功能 | 配置 |
|------|------|------|
| ADC | 音频采集 | 16kHz 采样率，单通道，DMA 乒乓缓冲 |
| LCD | 显示结果 | 类别名 + 置信度柱状图 |
| LED | 状态指示 | 蓝=待机 / 紫=采集中 / 绿=识别完成 / 红=错误 |
| KEY1 | 触发识别 | 按下开始采集并识别 |
| UART | 调试输出 | 发送识别结果和置信度到 PC 串口助手 |

### 8.5 主循环设计

```
上电
→ 初始化外设（ADC, LCD, LED, KEY, UART）
→ initialize_key_word_recognize()  // 预计算窗函数、Mel滤波器组、加载权重
→ LED 蓝（待机状态）

循环：
  KEY1 按下
  → LED 紫
  → ADC 采集 16000 采样点（1 秒）
  → result = key_word_recognize(waveform, 16000)
  → LED 绿
  → LCD 显示 label_dict[result]
  → UART 发送 "识别结果: %s, 置信度: %.2f"
  → 延时 2 秒
  → LED 蓝（回到待机）
```

## 十、思考题

1. **如何进一步降低 DSP 程序算力消耗？**
   - 使用 8bit 量化推理（BC-ResNet 原生支持）
   - 减少 Mel 滤波器数量（如 40 → 20）
   - 减少帧数（增大帧移 FRM_LEN）
   - 将部分 double 运算改为 float

2. **如何加快识别器响应速度？**
   - 减少采集时长（用 0.5 秒音频 + padding）
   - 使用流式推理，不等 1 秒采完即开始处理
   - 利用 DMA 双缓冲实现采集与推理并行

3. **如何改进识别方法，使其仅对坐在驾驶员位置的人有响应？**
   - 使用多麦克风阵列 + 波束形成定位声源方向
   - 仅在驾驶员方向上的语音激活识别

4. **如何改进识别方法，使其仅对若干常乘客有响应？**
   - 加入说话人验证（Speaker Verification）模块
   - 利用声纹特征做身份确认

5. **如何附加一定智能对话功能？**
   - 引入小型 LLM 或规则引擎做意图理解和多轮对话
   - 结合 TTS 实现语音反馈
